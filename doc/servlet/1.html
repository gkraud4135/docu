1. 서블릿 이란?

멀티 쓰레딩에 의해 사용자 요구를 처리하고 가공 하여 결과를 클라이언트 내보낸다.

 

JSP는 페이지의 요청이 있을 시에 최초에 1번 자바코드로 변환된 후 서블릿 클래스로 컴파일 된다.

즉, 실행시 JSP는 서블릿으로 변환된다.

 

서블릿과 JSP는 상호 연계되어 JSP에서 정적인 부분을 담당(사용자 뷰(view),

서블릿은 보다 동적인 처리를 위한 부분으로 사용(사용자 뷰와 프로그램 로직 사이클을 제어해 주는 역할).

 

2.HTTP프로토콜

 

1) HTTP프로토콜 : 비연결과 비상태의 특징을 갖는 프로토콜

웹 브라우져는 HTTP프로토콜에 맞게 요청(request)을 웹서버에 전송하고,

웹 서버는 요청에 따른 응답(reply)을 HTTP프로토콜에 맞게 웹 브라우저에 전송한다.

 

(1) 비연결(Connectionless) :클라이언트의 요청에 응답한 후 바로 연결 끊음

 

(2) 비 상태(Stateless) : 서버의 상태가 어떤지 간에 상관없이 요청을 함.

 

HTTP메소드는 여러가지가 있으나 거의 99.9% get과 post만 쓰임.

메소드 타입을 명시하지 않으면 디폴트는 get이다.

 

(1) Get :

- HTTP header에 정보를 실어 보냄.

- Start-line의 URL뒤에 붙는다.

-  적은양의 데이터 전송에 좋고 전달 속도 빠르지만 256byte가 한게이다.

 

(2) Post :

- HTTP의  body에 정보를 실어보냄.

- 데이터 사이즈에 제한이 없다.

 - 보안에 좋다.

 

 

3.서블릿 동작원리

서블릿은 javax.servlet패키지와 javax.servlet.http의 클래스와 인터페이스를 이용해서 만든다.

 

1) 서블릿의 종류

(1)Generic서블릿 :

 

 

 

 

 

 

 javax.servlet패키지는 프로토콜에 독립적인 서블릿을 만들기 위한 클래스를 제공한다.

 - 프로토콜에 독립적인 서블릿을 만들려면 Generic서블릿을 상속

 - Generic서블릿은 요청을 처리하기 위해 자신의 service()메소드를 오버라이드해야한다.

 - request(요청), response(응답)에 대한 접근을 제공

 

(2)Http서블릿

 

 

 

javax.seervlet.http패키지는 HTTP프로토콜의 고유한 기능(get, post)를 제공하는 서블릿을 만드는 클래스를 제공한다.

 

- doGet()메소드(Get요청시), doPost()메소드(post요청시)을 오버로드 한다.

 

- HttpServletRequest, HttpServletResponse는 HTTP요청과 응답에 대한 기능을 제공 한다.

 

2) 서블릿 요청과 응답의 과정

 

 

(1) Invokoer를 실행한다. (Thread를 만든다.(작업용 request당 1개))

 

(2) 생성된 Thread에서 service메소드(response(응답)내용 담김) 을 호출.

             (이는 쓰레드의 run메소드와 유사하다. )

 

 

(5) 결과를 클라이언트에게 전달한다. 이때 결과를 mime type으로 보내는데 웹 프라우저의 mime type은 text/html이다

 

Container는  tomcat, resine등은 웹 컨테이너이다..

 

3) 서블릿의 요청과 응답의 과정에서 컨테이너가 하는 일.

   (1) 객체가 있는지 유무 판단, 없으면 생성

   (2) Thread생성

   (3) service()호출

   (4) service() (HttpServlet class의 메소드)가  doGet()자동 호출

 

 

4. 서블릿 라이프 사이클

 

 

 

 

 

 

(1) 서블릿 태어남..(서블릿 로드됨)

(2) init() : 자격부여 (서블릿을 서비스 하기 위해 필요한 초기화 작업 수행)

 

             service()메소드 호출

(3) service(): request당 1개씩 호출. reload도 해당.

(4) destroy : 더 이상 서비스 하지 않을 경우에 서블릿 엔진에 의해 메모리에서 언로드 된다. 이때 destroy()메소드 실행

 

    destory()메소드는 아래와 같은 3가지 경우에 1번만 호출

     - container가 종료될때,

     - timeout이 되었을때,

     - class의 내용이 변경되었을때(재 컴파일시)될때
